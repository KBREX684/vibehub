更新日期：2026-04-19  
状态：Draft for final strategic freeze

## 0. 这是最终章

v11.0 的目标不是继续扩展 VibeHub 的能力边界，而是为未来 12 个月做一次明确、可执行、可冻结的战略收口。

本 RFC 一旦通过，意味着：

- 不再做新的主线叙事
- 不再在 6 个月内进行第二次方向性 pivot
- 不再同时维护两条以上一级产品故事
- 不再为了“看起来更完整”恢复已冻结的社区、撮合、团队协作、IDE、repo、通用智能体平台路线

### 0.1 冻结条款

通过本 RFC 后，以下事项默认冻结 12 个月：

- 团队协作 / 多人 Workspace 主线重启
- 项目发现 / Discover 主线重启
- 协作撮合 / Matching / 私信 / 聊天
- 内容流 / 帖子 / 排行榜 / 合集
- IDE / repo / CI / PR / Web IDE
- 通用智能体平台 / Agent 商店
- 海外支付 / 国际化恢复
- “再找一个更大赛道”的新战略提案

### 0.2 解冻条件

只有同时满足以下条件，才允许在 12 个月内解冻冻结项：

1. v11 的三个北极星指标连续两个自然月达标  
2. 新方向必须是对 `Studio -> Ledger -> Card` 单链路的增强，而不是平行第二主线  
3. 工程负责人、产品负责人、合规负责人共同签字  

对于单人项目，上述三方可由同一决议人签署三次，但必须保留书面记录。

## 1. 走过的路：客观复盘

### 1.1 v8：五段并列的诱惑

v8 的核心问题不是“功能不够多”，而是“主线过多”：

- 社区
- 项目展示
- 团队协作
- 开发者入口
- Agent 能力

这些功能单独看都成立，但并列放在同一产品里时，用户无法在 30 秒内理解 VibeHub 到底是什么。

### 1.2 v9 / v10：收口为 Workspace 的收益与问题

v9 / v10 的收口是有效的，因为它至少完成了两件事：

- 明确了“不做 IDE、不做 repo 主路径”
- 把产品收拢到 `Project -> Intent -> Workspace -> Snapshot -> Deliverable -> Agent`

但 v10 仍有两个难以回避的问题：

- 赛道正面开始接近 `Linear / GitHub Agents / Factory / Dify / CodeBuddy / Trae`
- 对小团队协作的价值虽然清晰，但推广成本仍然偏高，需要教育用户“为什么不用现有工具组合”

### 1.3 v10.5：为什么会出现“公证人方向”

从工程现实看，VibeHub 已经沉淀了几个非常适合转向“可信交付证明”的资产：

- `Workspace`
- `SnapshotCapsule`
- `WorkspaceDeliverable`
- `AgentTask`
- `AgentConfirmationRequest`
- `AuditLog`

这些对象天然比“社区”更像一条证据链。  
也就是说，VibeHub 现在最像的，不再是“团队协作平台”，而是“能把 AI 工作过程留下证据的系统”。

### 1.4 三条候选路的四维评分

评分标准：

- 差异化：是否能避开国内正面红海
- 护城河：是否有复合型能力，而不是 UI 即可复制
- 落地成本：是否能复用现有代码主干
- 推广清晰度：是否能一句话讲明白

| 路线 | 差异化 | 护城河 | 落地成本 | 推广清晰度 | 总分 |
| --- | --- | --- | --- | --- | --- |
| A. 继续做 Team Workspace 协作中枢 | 2 | 3 | 5 | 2 | 12 |
| B. 改造成可信工作账本（Studio / Ledger / Card） | 5 | 4 | 4 | 4 | 17 |
| C. 改造成撮合 / 外包 / 匹配网络 | 2 | 1 | 2 | 3 | 8 |

### 1.5 为什么最终选“可信工作账本”

最终不建议把 v11 的公开定义写成“操作公证账本”，原因很简单：

- “公证”在中文语境下带有强法律承诺色彩
- 并非每一条记录都会天然获得司法公证效力
- 真正的产品价值并不只在“上链”或“时间戳”，而在“工作过程可追溯”

因此，v11 推荐采用更稳的产品定义：

**对外定义：可信工作账本**  
**内部实现：Studio -> Ledger -> Card**  
**可选增强：时间戳 / 水印 / 区块链 / 第三方存证**

## 2. 一句话定位（冻结）

### 2.1 对外定位

**VibeHub — 你和 AI 一起完成的工作，有据可查。**

### 2.2 对内定义

VibeHub 是面向中国独立开发者、一人公司与微型工作室的 **AI 工作留痕与交付证明系统**。  
它不负责替代 IDE、repo、IM 或项目管理工具，而负责把“你到底做了什么、AI 参与了什么、客户最终能看到什么”沉淀为可展示、可导出、可验证的工作账本。

## 3. 用户与场景（冻结）

### 3.1 唯一目标人群

v11 只服务一类核心用户：

- 中国独立开发者
- 一人公司 / One-Person Company（OPC）创始人
- 2 人以内的微型 AI 工作室

不再优先服务：

- 大型企业研发组织
- 10 人以上团队
- 泛内容社区创作者
- 寻找“另一个 GitHub / Cursor / Dify”的用户

### 3.2 高频场景：在 Studio 里和 Agent 干活

用户不会在 VibeHub 里写全部代码。  
用户仍在本地 IDE、设计工具、脚本工具中工作；VibeHub 的 `Studio` 负责：

- 记录工作单元
- 挂载关键文件和快照
- 记录 Agent 行为与人工确认
- 输出一条可回看的工作账本

### 3.3 中频场景：给客户导出月度 / 周度 Ledger

这是 v11 的关键付费触点。  
客户不需要理解你的 IDE 历史，但会在意：

- 本周做了什么
- 哪些部分使用了 AI
- 交付是否可回溯
- 风险与声明是否写清楚

### 3.4 低频场景：更新 Trust Card / 申请扶持 / 接单自证

用户需要一个公开页，不是为了“社交”，而是为了：

- 给客户一个可信主页
- 给平台一个合规 / 可信说明
- 给自己一个“不是 PPT，而是有证据的能力页”

## 4. 三件事架构

v11 只保留三件事，且三者必须形成单向因果链。

### 4.1 Studio（做）

`Studio` 是工作发生的地方，但它不是 IDE。

它是：

- 项目容器
- 工作时间线
- 文件与快照入口
- Agent 任务与确认入口
- 交付审核入口

它不是：

- 在线编辑器
- 代码托管仓库
- 团队 IM
- 通用任务平台

### 4.2 Ledger（签）

`Ledger` 是 v11 的核心资产。  
它记录的不是“聊天记录”，而是**工作证据**：

- 谁做了什么
- 什么时候做的
- 关联了哪些文件 / 快照 / 交付包
- 是否有 AI 参与
- 是否有人工确认
- 是否做过第三方存证或时间戳

### 4.3 Card（晒）

`Card` 是公开输出层。  
它不是“作品广场”，而是一个可信说明页，展示：

- 你是谁
- 你最近完成了哪些真实项目
- 你的 AI 使用边界
- 你的工作账本摘要
- 你的可信度指标

### 4.4 三者因果

v11 的唯一主链为：

**Do -> Sign -> Show**  
**Studio -> Ledger -> Card**

没有第四件一级事。

### 4.5 不在主线

以下对象即使保留兼容数据，也不再属于产品主线：

- Team
- Discover
- Collaboration Intent
- Notifications 多分类系统
- Posts / Discussions / Collections / Leaderboards

## 6. 数据层（最终）

### 6.1 新增 4 张核心表 + 1 个缓存层

推荐新增：

1. `LedgerEntry`
   - 工作账本的最小记录单元
   - 关联 Studio、Snapshot、Deliverable、AgentTask、Artifact

2. `AigcStamp`
   - 记录某个交付物或导出内容的 AIGC 标识状态
   - 包含显式标识、隐式标识、导出配置

3. `OpcProfile`
   - 一人公司 / 独立开发者的公开可信档案
   - 作为 Trust Card 的公开资料源

4. `LegalAttestationLink`
   - 记录第三方时间戳、区块链存证、公证或其他司法关联凭据

不建议把 `OpcTrustMetric` 在 v11.0 初期作为主数据表。  
更合理的方式是：

- 先作为派生指标计算层
- 若后续需要缓存，再引入物化表或每日聚合表

### 6.2 升级既有表

建议升级并复用：

- `Workspace`：概念重命名为 `Studio`
- `SnapshotCapsule`：作为 Ledger 的证据源之一
- `WorkspaceDeliverable`：作为交付审核核心对象
- `WorkspaceArtifact`：作为附件与导出物源
- `AgentTask`：作为 AI 工作记录源
- `UserSubscription`：收口为 Free / Pro 两档

### 6.3 锁创建对象

以下对象进入 read-only / archive 模式：

- `Team`
- `TeamMembership`
- `TeamAgentMembership`
- `CollaborationIntent`

### 6.4 不做删除

v11 不要求删旧表。  
理由：

- 保留数据完整性
- 降低迁移风险
- 保留审计链
- 允许未来只读回看历史阶段

## 10. 推广策略（修正版）

原草案中的“fear-based + 罚款上限”方向需要修正。

### 10.1 为什么不能把“罚 1000 万”作为主宣传语

虽然《人工智能生成合成内容标识办法》与配套标准 **GB 45438-2025** 已于 **2025 年 9 月 1 日**施行，且监管对标识违规已有真实执法案例，但：

- 罚则来源并不只来自单一标准本身
- 不同主体承担的义务并不完全相同
- “个人接单开发者一定会因未标识被罚 1000 万”这种说法不严谨，法律风险高

所以，v11 不能把夸张罚则当增长主文案。

### 10.2 正确的推广逻辑

v11 推荐采用三段式价值主张：

1. **客户信任**
   - “我用了 AI，但我能讲清楚我做了什么。”

2. **交付防扯皮**
   - “不是口头解释，而是有时间线、有快照、有导出记录。”

3. **合规准备**
   - “AIGC 标识、导出声明、第三方锚定都可选开启。”

### 10.3 自带传播点

每张 `Trust Card` 都是传播页。  
每次 `Ledger` 导出都是一个外发触点。  
这比做 Discover 广场更适合小产品。

### 10.4 渠道

推荐优先：

- 小红书
- 即刻
- V2EX
- 程序员客栈

### 10.5 内容角度

不要发“我们是下一代平台”。  
要发：

- AI 接单如何防扯皮
- 客户问“你是不是全靠 AI 做的”怎么回答
- 如何把交付过程变成可信记录
- 如何给作品和过程同时加说明

## 11. 90 天 PMF 验证

### 北极星 1：账本启用率

注册后 7 日内，开启 Studio 工作账本的用户占比 >= 60%

### 北极星 2：导出率

月度 Ledger 或 Trust Card 导出率 >= 25%

### 北极星 3：Pro 转化率

活跃用户中 Pro 转化率 >= 5%

### 辅助指标

- Trust Card 完整度 >= 60% 的用户占比
- 平均每个活跃用户每月新增 LedgerEntry 数
- 带 AIGC 标识的导出占比
- 首次创建 Studio 到首次导出 Ledger 的中位时间

## 14. 为什么这条路值得冻结 12 个月

### 14.1 国内竞品并不集中在这个产品层

国内现有产品主要分布在三类：

1. **代码托管 / DevOps / 企业研发管理**
   - 代表：Gitee

2. **AI IDE / 设计到部署一体化**
   - 代表：Trae、CodeBuddy

3. **通用 AI 工作台 / 模板市场**
   - 代表：Dify、扣子空间

而 v11 选择的不是上述任一红海层，而是：

**独立开发者的 AI 工作证据层。**

### 14.2 国外有支点，但没有完全打到这个组合

国外已有两类强信号：

- `Adobe Content Credentials`：内容来源与生成方式可验证
- `OpenAI / Sora`：可见 + 隐式 provenance signal

但它们主要是：

- 内容真实性基础设施
- 平台级 provenance 机制

还没有把以下三件事打成一个面向独立开发者的产品：

- 工作过程记录
- 客户交付证明
- 中国本地合规锚定

### 14.3 国内基础设施已成熟，但产品层仍空

国内已经有：

- 阿里云水印与 AIGC 标识能力
- 至信链、易保全、联合信任时间戳等存证能力

但它们是基础设施，不是面向独立开发者的完整产品工作流。  
这意味着 v11 的机会不在“发明水印”，而在“把这些能力组织成一个有 PMF 的产品”。

### 14.4 与现有代码主干的贴合度高

v11 不是从零开始，它能复用当前主干：

- `Workspace` -> `Studio`
- `SnapshotCapsule` -> Ledger 证据块
- `WorkspaceDeliverable` -> 交付审核
- `WorkspaceArtifact` -> 附件 / 导出物
- `AgentTask + AgentConfirmationRequest` -> AI 工作与人工确认链

所以它比重新回去做社区、撮合、团队平台要更便宜。
