# VibeHub v11.0 — 后端任务计划书（GPT 执行）

> 配合 RFC：`docs/v11.0-final-chapter-rfc.md`
> 适用对象：GPT（后端工程师角色）
> 使用方式：把每个 P 阶段对应的 prompt 整段复制给 GPT，让它先用 plan 模式拆解，再执行。
> 仓库结构假定：Next.js + Prisma + PostgreSQL，代码在 `web/`。

## 3. P1 — Ledger 数据层

### 3.1 给 GPT 的完整 prompt

```text
任务 ID：[v11/P1][BE] LedgerEntry schema + 哈希链 + API + 回填

目标：
  落地操作公证账本的数据层，包含 schema、签名链、API、历史回填。
  这一阶段是 v11 的核心，所有后续阶段都依赖它。

数据契约（先冻结）：

  // Prisma model
  model LedgerEntry {
    id              String   @id @default(cuid())
    workspaceId     String
    actorType       LedgerActorType                   // user | agent
    actorId         String
    actionKind      String                            // workspace.artifact.upload 等
    targetType      String?
    targetId        String?
    payloadHash     String                            // SHA-256(JSON.stringify(payload))
    prevHash        String?                           // 上一条同 workspaceId 的 signature；首条为 null
    signature       String                            // base64 ed25519 sig of (payloadHash + '|' + (prevHash ?? ''))
    signedAt        DateTime @default(now())
    anchorChain     LedgerAnchorChain @default(vibehub)
    anchorTxId      String?
    anchorVerifiedAt DateTime?
    @@index([workspaceId, signedAt])
    @@index([actorType, actorId, signedAt])
  }
  enum LedgerActorType { user agent }
  enum LedgerAnchorChain { vibehub zhixin baoquan }

  // 既有表加字段（仅加，不删）
  AgentActionAudit:        + ledgerEntryId String?
  AgentConfirmationRequest: + ledgerEntryId String?
  SnapshotCapsule:         + ledgerEntryId String?
  WorkspaceDeliverable:    + ledgerEntryId String?
  Workspace:               + ledgerEnabled Boolean @default(true)

  // 5 个 API（路径 + 响应类型）
  GET  /api/v1/me/ledger?workspaceId=&from=&to=&actor=&kind=&cursor=&limit=
       → { items: LedgerEntry[], cursor: string | null }
  GET  /api/v1/workspaces/:id/ledger?...
       → 同上，权限：仅 workspace owner
  GET  /api/v1/workspaces/:id/ledger/export?format=json
       → application/json，下载文件 ledger-{workspaceId}-{from}-{to}.json
       → 文件结构 { signedBy: 'vibehub-signer-v1', publicKey: '...', entries: LedgerEntry[] }
  GET  /api/v1/ledger/:id/verify
       → 公开，返回 { ok: boolean, brokenAt: string|null, totalChecked: 1, signedBy: '...' }
  POST /api/v1/ledger/verify-bundle
       → 公开，body=ledger.json，返回 { ok, brokenAt, totalChecked, signedBy }

  // 错误码
  LEDGER_DISABLED          (workspace.ledgerEnabled=false)
  LEDGER_HASH_BROKEN       (验证失败时)
  LEDGER_NOT_FOUND
  LEDGER_FORBIDDEN

执行步骤（先 plan，再做）：

  1. 新建 web/src/lib/ledger/signer.ts
       - 从 env VIBEHUB_LEDGER_SIGNER_PRIV (base64 ed25519 私钥) 加载
       - 暴露 publicKey() / sign(message: string): string
       - 本地开发用 sample key（在 .env.example 给出 dev key）
       - 私钥不存 DB

  2. 新建 web/src/lib/ledger/hash-chain.ts
       - computePayloadHash(payload: unknown): string  // SHA-256 hex
       - composeSigningMessage(payloadHash, prevHash): string
       - verifyEntry(entry: LedgerEntry, prevSignature: string|null, publicKey: string): boolean
       - verifyBundle(entries: LedgerEntry[], publicKey: string): { ok, brokenAt, totalChecked }

  3. Prisma migration
       - 文件名 web/prisma/migrations/<timestamp>_v11_p1_ledger/migration.sql
       - 包含上述 schema 改动

  4. 新建 web/src/lib/repositories/ledger.repository.ts
       方法：
         appendEntry({ workspaceId, actorType, actorId, actionKind, targetType, targetId, payload }): Promise<LedgerEntry>
           - 在事务里读最近一条 entry 的 signature 作为 prevHash
           - 计算 payloadHash → signature
           - INSERT
         listByWorkspace({ workspaceId, from, to, actor, kind, cursor, limit })
         listByUser(...)
         getById(id)
         exportJsonBundle({ workspaceId, from, to })
         verifyBundle(bundle)

  5. 5 个 API route 文件（按上述路径建）
       - 鉴权使用 web/src/lib/auth.ts 现有方法
       - 公开 endpoint（verify / verify-bundle）不需要登录，但要加全局速率限制

  6. OpenAPI 注册
       - web/src/lib/openapi-spec.ts 增加上述 5 条
       - 生成 web/src/lib/generated/api-types.ts （这是共享文件，PR 描述里
         明确写 "@PM 需手动 merge：web/src/lib/generated/api-types.ts"）

  7. 历史回填脚本 web/scripts/v11-backfill-ledger.ts
       - 按 createdAt 升序遍历 AgentActionAudit / AgentConfirmationRequest /
         SnapshotCapsule / WorkspaceDeliverable
       - 对每一条调 ledger.repository.appendEntry
       - 把回写的 ledgerEntryId 更新到原表
       - 支持 --dry-run / --workspace=ID 参数
       - npm script: "v11:backfill-ledger": "tsx scripts/v11-backfill-ledger.ts"

  8. 在 v10 已有的写入路径上"埋钩子"：
       - 凡是 AgentActionAudit / SnapshotCapsule / WorkspaceDeliverable / AgentConfirmationRequest
         创建处，调 ledger.repository.appendEntry 写入对应 entry，并把 ledgerEntryId
         回写到原表。
       - 这是关键步骤：以后所有新数据都有 ledger，老数据由步骤 7 回填。

  9. 测试 web/tests/v11-ledger-hash-chain.test.ts
       - 用 ledger.repository 连续追加 10 条
       - verifyBundle 通过
       - 篡改第 5 条 payload，再 verify → ok=false, brokenAt=第 5 条 id
  10. 测试 web/tests/v11-ledger-api.test.ts
       - GET /me/ledger 鉴权 + 分页
       - GET /workspaces/:id/ledger 权限隔离
       - GET /workspaces/:id/ledger/export 返回正确 bundle
       - GET /ledger/:id/verify 公开可达
       - POST /ledger/verify-bundle 公开可达 + 参数校验

不要做：
  - 不要做 PDF 导出（P4 才做）
  - 不要做司法锚定调用（P6 才做）
  - 不要碰前端 fixture

验收：
  - npm run prisma:migrate
  - npm run prisma:generate
  - npm test --workspace=web -- v11-ledger
  - 在 dev DB 上 npm run v11:backfill-ledger -- --dry-run，看到统计
  - 跑一次正式回填，再跑 verify-bundle 全部 ok

PR 标题：[v11/P1] LedgerEntry + hash chain + API + backfill
PR 描述贴出：schema diff、5 个 API 路径、回填统计、关键测试输出。
```
