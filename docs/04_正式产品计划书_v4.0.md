## 0. 文档说明

本计划书由 MVP 代码审计（见第一章）驱动，取代原 v3.0 中 KPI 验证导向的阶段门槛，改为以**功能完整性与技术严格实现**为封版条件。每阶段均附 AI 模型分工表，指导多模型协作开发流程。

## 一、MVP 代码逐单元审计

### 1. 认证与会话系统

**现状**
- 认证入口仅为 `GET /api/v1/auth/demo-login?role=admin|user`，本质是硬编码 demo 用户
- 会话 Token = Base64(payload) + HMAC-SHA256 签名（payload 明文可读，无加密）
- 无用户注册、登录、密码管理、邮件验证、OAuth 任何真实认证流
- `secure` cookie 仅在 `NODE_ENV=production` 时启用，本地开发存在 CSRF 风险

**与计划书差距**
- 计划书 §7.3 要求"权限模型（用户、创作者、团队管理员、企业观察者）"，目前仅有 `guest/user/admin` 三角色，创作者角色、企业观察者角色均未实现
- 无任何真实身份体系，无法支撑正式产品的注册/登录/账户管理

**问题等级：P0 阻塞**

---

### 2. 用户与创作者资料系统

**现状**
- `User` 和 `CreatorProfile` 模型结构完整（Prisma schema 正确）
- 无任何创作者资料创建/编辑 API 端点
- 无用户注册 API
- 前端无"创建创作者主页"、"编辑个人资料"页面
- `toCreatorDto` 中 `collaborationPreference` 三档映射逻辑正确

**与计划书差距**
- 计划书 §3.2 要求"创作者页：能力画像、代表作、协作偏好、可投入时间"，目前只读，无写入路径

**问题等级：P0 阻塞**

---

### 3. 项目画廊

**现状**
- `GET /projects`、`GET /projects/:slug`、`GET /projects/facets` 实现完整
- 发现页 `/discover` 支持 `query/tag/tech/status/team` 过滤器
- 无项目创建/编辑 API 和 UI（`PATCH /projects/:slug` 仅限绑定团队）
- 项目数据全部来自 seed，用户无法自行提交项目
- 计划书要求的"里程碑、关键指标"字段未在 `Project` schema 中建模

**与计划书差距**
- 计划书 §3.2 要求项目页含"Demo、里程碑、关键指标"，schema 缺少项目级 `milestones` 和 `metrics` 字段
- 无投资者视图的专属角色门控

**问题等级：P1 重要**

---

### 4. 讨论广场

**现状**
- `GET/POST /posts`、`GET /comments`、`POST /comments` API 实现
- 内容审核流程（pending → approved/rejected）完整
- 前端无帖子创建表单，无 Markdown 编辑器，无图片上传
- 无"引用回复"、"收藏"、"关注"、"精华"机制
- 帖子排序仅按 `createdAt`，无热门/精华加权

**与计划书差距**
- 计划书 §3.2 明确要求"收藏、关注、主题标签、精华评审"，均未实现
- P2 对照表已承认"精华机制未单独建模"，这是未完成项

**问题等级：P1 重要**

---

### 5. 协作意向

**现状**
- `join/recruit` 双类型意向提交完整
- 管理端审核、漏斗指标 API 完整
- 前端 `CollaborationIntentForm` 存在但仅为组件，未嵌入任何 SSR 页面
- 无协作意向的"配对成功"下游通知/行动路径

**与计划书差距**
- 计划书 P2 要求"协作意向配对成功率"，目前意向批准后无任何后续行动触发

**问题等级：P2 待完善**

---

### 6. Team 协作空间

**现状（最完整的模块）**
- Team CRUD、成员申请审批、任务看板（3列）、里程碑、任务↔里程碑关联、RBAC、站内通知 均已实现
- Kanban 重排逻辑仅支持同列相邻交换（非拖拽跨列，明确文档化）
- `TeamTask.assigneeUserId` 存在但无"我的任务"聚合视图
- 无团队级讨论/协作日志模型
- `Team.mission` 为 optional，无团队介绍富文本
- 信誉系统（计划书 P3 核心交付物）完全未建模

**与计划书差距**
- 计划书 §3.2 P3 要求"信誉系统：贡献记录、完成率、协作评分"，`schema.prisma` 无任何相关模型
- 计划书 P3 要求"商业化首发：Pro 会员、团队增值包"，完全未启动

**问题等级：P2 待完善（信誉/商业化 = P0 for P3）**

---

### 7. API 密钥与 MCP

**现状（最规范的模块）**
- SHA-256+pepper 哈希存储，明文仅展示一次，安全设计合理
- 作用域系统完整（`read:public`、`read:projects:list` 等）
- 内存 + Redis 双路限流，降级策略正确
- MCP v2 带审计日志，`McpInvokeAudit` 无 FK 约束（应用层保证）
- OpenAPI 3.0 + CI 验证 + 必选路径白名单

**问题**
- `McpInvokeAudit` 无 Prisma 外键 → 用户删除后审计记录孤立（合规风险）
- `ApiKey.scopes` 存储为 JSON string[]，无类型约束（应改为 `String[]` 并在 DB 层强制）
- MCP v2 invoke 为 HTTP JSON，非 stdio MCP 标准协议，与真实 MCP 客户端不兼容

**问题等级：P2 待修复**

---

### 8. `repository.ts` 架构

**现状**
- 单文件 4300+ 行，同时包含 mock 路径和 Prisma 路径
- `USE_MOCK_DATA` 环境变量在运行时分支，存在生产误配置风险
- mock 数据与业务逻辑耦合，影响测试隔离

**问题等级：P1 技术债**

---

### 9. 前端质量

**现状**
- `globals.css` 实现基础布局，无完整设计系统
- 页面无 loading state / error boundary / skeleton
- 列表无分页 UI（API 支持分页但前端未使用）
- 无响应式/移动端适配
- 首页 hero 写死"P1 MVP"文案，需正式化

**问题等级：P1 重要**

---

### 10. 运维与可观测性

**现状**
- `/api/v1/health` 仅返回 `{ status: "ok" }`，无 DB 连通性检查
- 无结构化日志（无 pino/winston）
- 无错误监控（无 Sentry/Bugsnag）
- 无性能追踪
- 无后台任务队列（邮件发送在请求链路内同步执行）

**问题等级：P1 重要**

---

### 11. 合规与安全

**现状**
- 无 ICP 备案、隐私政策、服务条款页面
- 无内容举报处理 SLA 机制（模型存在但无 UI）
- AIGC 生成内容标识未规划
- 无未成年人保护机制

**问题等级：正式上线必须项**

## 三、产品定位与核心架构目标

### 产品定位（不变）

VibeHub：面向中文 VibeCoding 生态的**讨论广场 + 项目画廊 + 协作平台 + Agent 数据基础设施**一体化平台。

北极星目标：在中文 VibeCoding 领域，成为"人类创作协作网络 + AI Agent 可调用知识基础设施"的事实标准入口。

### 核心架构目标

- **数据完整性**：每个功能闭环，用户可完成端到端操作（创建、展示、协作、发现）
- **真实身份体系**：有效账户注册 → 个人资料 → 公开页面的完整路径
- **工程可维护性**：模块化 repository、类型安全 API、自动化质量门禁
- **可观测生产部署**：结构化日志、错误监控、健康检查、异步任务队列

### 技术栈选型（延续 MVP + 补充）

| 层 | 技术 | 说明 |
|---|---|---|
| 框架 | Next.js 15 App Router | 延续 |
| UI | React 19 + Tailwind 4 | 延续 |
| 设计系统 | **shadcn/ui**（Radix UI）| 替换手写 globals.css |
| 数据库 | PostgreSQL + Prisma 6 | 延续 |
| 认证 | **NextAuth.js v5** | 替换 demo-login |
| 富文本 | **Tiptap** | 新增，帖子/项目描述 |
| 文件存储 | **Cloudflare R2 / S3** | 新增，头像/截图 |
| 全文搜索 | PostgreSQL `tsvector`（P1）→ **Meilisearch**（P3）| 渐进升级 |
| 向量检索 | **pgvector** | P3 新增，语义推荐 |
| 后台任务 | **BullMQ + Redis** | 新增，异步邮件/通知 |
| 日志 | **Pino** 结构化日志 | 新增 |
| 错误监控 | **Sentry** | 新增 |
| 支付 | **Stripe** | P3 新增，订阅 + 一次性 |
| 缓存/限流 | ioredis | 延续 |
| 测试 | Vitest | 延续 |
| CI | GitHub Actions | 延续 |

## 六、商业模式

### 收入来源

1. **个人 Pro**：高级检索、深度分析、优先曝光、协作增强
2. **Team Pro**：团队协作空间高级能力与管理能力
3. **API 收费**：面向第三方工具和 Agent 调用，按量计费
4. **企业席位**：项目与人才发现、关注列表、尽调面板
5. **服务收入**：主题挑战赛联办、品牌合作、精选生态报告

### 定价原则

1. 低门槛试用，高价值功能付费
2. 个人付费强调"效率与曝光"，企业付费强调"筛选效率与可信度"
3. API 与数据产品采用分层计费与配额机制

---

## 七、合规要求（中国境内）

1. ICP 备案与平台信息服务合规（上线前必须）
2. 生成式 AI 服务相关规范（涉及 AIGC 输出时）
3. 深度合成与算法推荐相关规范（涉及推荐分发时）
4. 个人信息保护、数据安全、未成年人保护义务
5. 内容治理：违规识别、分级处置、申诉机制（上线前必须）
6. 数据可携带导出 API（`GET /api/v1/me/export`，P4 交付）

## 八、各阶段关键风险

| 阶段 | 最大风险 | 应对策略 |
