# VibeHub 生产化升级总计划 v7.0（开发者社区优先版）

> 规划日期：2026-04-16  
> 基于：v5.0 路线图 + P0-P4 已落地代码全量审计 + 新战略输入  
> 定位：中国国内落地优先的开发者社区 + 项目展示 + 团队协作 + Agent 接入生态  
> 状态：**纯规划文档，不涉及任何代码修改**

---

## A. 战略结论

### 方向正确，应纳入规划

1. **中国国内落地优先 + 合规前置** — 方向完全正确。当前登录体系仅有 GitHub OAuth，邮箱登录缺失是硬伤。隐私政策、用户协议、内容审核框架是中国上线的前提。凡涉及具体法规条款的实施细节（实名制边界、数据跨境、日志留存期限），均标记"需专项合规核验"。
2. **GitHub 降为辅助层** — 正确。当前 GitHub 是唯一登录通道，对中国用户是致命障碍（国内 GitHub 访问不稳定，部分用户无 GitHub 账号）。GitHub 保留为关联能力，不再是唯一入口。
3. **社区三大内容核心强化（项目画廊、广场讨论、团队协作）** — 完全正确，与既有主线一致。先收口主链路，再做推荐流等高复杂度扩展。
4. **支付适配中国路径（支付宝/微信支付）** — 必须做。当前仅绑定 Stripe，对中国用户几乎不可用。P0 完成支付抽象层设计，P1 完成正式业务流接入。
5. **管理员后台增强 + AI 审核助手** — 方向正确。当前后台仅覆盖 `/admin/collaboration`、`/admin/moderation`、`/admin/users`，远不够生产运营需要。AI 辅助审核严格限定为"建议型"。
6. **企业能力继续弱化** — 正确。当前 `EnterpriseProfile` 模型和验证流程已存在但过重。降级为"认证标识 + 资料字段"，不再作为产品中心。

### 需要降级

7. **Agent 协作闭环** — 方向极有吸引力，但复杂度极高。用户-agent-团队-任务-审查的全链路闭环涉及身份模型、权限继承、审计机制、通知流转等多个未验证维度。**必须拆期：P1 做基础绑定模型，P2 做角色体系与协作流，不得直接上 P0**。
8. **广场推荐流/热度流/关注流/时间流** — 四种 feed 流对内容分发至关重要，但算法推荐涉及数据积累和调参周期。**P0 只做时间流 + 基础排序，P1 补齐推荐/热度/关注流**。

### 必须后移

9. **挑战赛/活动** — 明确后移至 P3。当前阶段不分配资源。代码中已有 `Challenge` 模型和 `/challenges` 页面，不删除但冻结，不做任何新功能开发。
10. **多 agent 自治编排** — P3 或更后期。在基础 agent 绑定模型和角色体系验证之前，任何自治编排都是空谈。

### 当前不该做

11. **企业工作台/投资者视图重建** — 企业能力已明确弱化，不得以任何形式重新膨胀。
12. **Light/Dark 主题切换** — 当前设计系统已统一为 Monochrome Geek，切换功能是锦上添花，不是生产阻塞项。
13. **PWA / Service Worker** — 中期能力，当前无需求支撑。

## B. 产品主飞轮重定义

### 一句定义

**开发者在广场讨论中产生灵感 → 在项目画廊中展示作品 → 在团队平台中组队协作 → 通过 API/MCP 让 agent 参与执行 → 协作成果反哺社区内容供给。**

### 解释

VibeHub 的唯一飞轮是"内容产生 → 项目沉淀 → 协作转化 → 成果回流"。四个核心模块的关系不是并列的，而是有明确的上下游：

- **广场讨论**（上游入口）：开发者发布想法、经验、求助，产生初始流量和互动黏性。是用户获取和留存的第一触点。
- **项目画廊**（中游枢纽）：优质讨论沉淀为结构化项目展示，形成可发现、可验证的开发者资产。是平台价值的核心载体。
- **团队协作**（下游执行）：项目吸引协作者，组队后在 team 空间中推进任务。是平台从"看"到"做"的关键转化。
- **Agent 接入**（底层能力层）：通过 API/MCP 让开发者的 agent 工具能够读取、操作平台数据，降低协作摩擦。是差异化壁垒，但不是飞轮的起点。

**关键约束：agent 接入是飞轮的加速器，不是飞轮本身。没有社区内容和协作场景，agent 接入毫无价值。因此 agent 能力必须后于社区核心建设。**

## C. P0 / P1 / P2 / P3 分级路线图

### P0：生产落地阻塞项

> 目标：消除中国国内上线的所有硬性阻塞，将平台从"可演示"状态推进到"可放量"状态。

#### P0-1：登录体系重构

- 新增邮箱 + 密码注册/登录（作为主登录方式）
- 邮箱验证码 / 邮箱确认流程
- GitHub OAuth 保留为"关联账号"能力，不再是唯一登录入口
- 已有用户的 GitHub 账号自动关联（基于 email 匹配）
- 登录页改造：邮箱登录为主入口，GitHub 登录为辅助入口
- 密码存储使用 bcrypt/argon2，不存明文
- 验收标准：用户可通过邮箱注册、登录、重置密码；GitHub 账号可关联/解绑；无 GitHub 账号的用户可完整使用平台

#### P0-2：企业能力降级

- `EnterpriseProfile` 降级为"认证标识"：保留企业名称、Logo、认证状态字段
- 移除企业工作台（`/workspace/enterprise`）的主导航入口
- 企业认证结果仅展示在用户个人资料页的徽章位
- `EnterpriseMemberInvite` 模型冻结，不再开发多席位功能
- 后台保留企业认证审核能力，但审核结果仅影响标识展示
- 验收标准：企业认证仅产出一个认证徽章；无企业专属功能入口；后台可审核认证申请

#### P0-3：Mock 退出生产路径

- 审计所有 `isMockDataEnabled()` 调用点（当前分布在 `repository.ts`、`repositories/*.ts`、`billing.repository.ts` 等 18+ 文件）
- 生产环境 `RUNTIME_MODE` 强制为 `production`，禁止 fallback 到 mock
- Mock 数据路径仅保留在 `development`/`test` 模式
- 启动时校验：production 模式下 `DATABASE_URL` 必须存在，否则拒绝启动（`env-check.ts` 已有雏形）
- 验收标准：`NODE_ENV=production` 下无任何 mock 路径可达；启动时自检通过

#### P0-4：中国基础合规框架

- 隐私政策页面（`/privacy` 已有路由占位，需填充内容）
- 用户协议/服务条款页面（`/terms` 已有路由占位，需填充内容）
- 用户注册时必须勾选同意隐私政策和用户协议
- 内容举报通道已有（`ReportTicket` 模型 + API），需确保前端可达
- 用户可删除自己的账号（数据清理）
- 标记"需专项合规核验"的项目：实名制边界、数据跨境处理、日志留存期限、AIGC 服务备案、ICP 备案
- 验收标准：隐私政策和用户协议页面可访问；注册流程包含同意勾选；举报通道可用；账号可注销

#### P0-5：API / 鉴权 / 密钥 / 限流 / 审计生产基线

- Session 服务端吊销能力（`User.sessionVersion` 已在 schema 中，确认全链路生效）
- CSRF 防护加固（`X-CSRF-Token` 双重检验，当前仅依赖 `SameSite=Lax`）
- Admin 路由边缘验证（`middleware.ts` 中 session decode + role 校验）
- API Key scopes 已有完整实现（`api-key-scopes.ts`），需审计所有 scope 边界
- 结构化日志系统（`logger.ts` 已存在，需替换所有 `console.*`）
- MCP invoke audit 已有（`McpInvokeAudit` 模型），确保写入完整
- 验收标准：敏感操作后 session 可失效；mutating API 有 CSRF 保护；非 admin 用户无法访问 admin 页面；关键操作有结构化日志

#### P0-6：项目画廊主链路收口

- 确认 `/projects/new` → `POST /api/v1/projects` 全链路闭环
- 实现 `/projects/[slug]/edit` → `PATCH /api/v1/projects/[slug]` 全链路
- 项目列表页基础排序：最新、最热（基于收藏/协作意向数）
- 项目详情页核心信息完整展示
- 验收标准：用户可创建、编辑、查看项目；项目列表支持基础排序

#### P0-7：广场讨论主链路收口

- 讨论列表：时间排序（已有）+ 热门排序（已有）
- 发帖、评论、回复、编辑、删除全链路已闭环（v1.1 审计确认 ✅）
- 帖子审核流（`reviewStatus`）已有，确认 admin 端操作通畅
- 验收标准：用户可发帖、评论、互动；admin 可审核帖子

#### P0-8：团队协作主链路收口

- 团队创建、入队申请与审批（已闭环 ✅）
- 任务板 CRUD（`TeamTask` 已有）
- 里程碑 CRUD（`TeamMilestone` 已有）
- 团队聊天（WebSocket + DB 持久化已有）
- `TeamRole` 当前仅 `owner | member`，收口时暂不扩展
- `/teams/[slug]/settings` 页面补全（当前缺失）
- 验收标准：团队创建到任务推进的完整链路可用

#### P0-9：支付抽象层设计

- 设计 `PaymentProvider` 接口（创建支付会话、处理回调、客户门户、取消/查询订阅）
- 当前 Stripe 实现（`billing/checkout`、`billing/portal`、`billing/webhook`）作为第一个 provider
- 预留支付宝/微信支付的 provider 接口
- `UserSubscription` 模型增加 `paymentProvider` 字段
- Free/Pro 订阅逻辑与支付渠道解耦
- 验收标准：支付抽象层代码就位；Stripe 路径不受影响；新增 provider 仅需实现接口

#### P0-10：管理员后台治理能力重构

- 当前后台仅有：`/admin/collaboration`、`/admin/moderation`、`/admin/users`
- 必须补充：
  - 企业认证审核面板（API 已有，需前端）
  - 举报工单管理面板（`ReportTicket` 已有模型和 API）
  - 审计日志查看面板（`AuditLog` 已有模型）
  - 系统健康状态面板（基础 metrics）
- 验收标准：admin 可管理用户、审核内容、处理举报、查看审计日志

#### P0-11：后台 AI 审核助手 MVP

- 角色定位：预审助手 + 摘要生成器，不是自动执法者
- MVP 能力：
  - 对举报工单自动生成摘要（被举报内容 + 举报理由 + 历史记录）
  - 对企业认证申请自动生成资料核验建议（非最终裁决）
  - 对帖子内容进行基础风险标记（敏感词检测 + 风险分级建议）
- 所有 AI 建议必须标记为"AI 建议"，管理员一键确认/驳回
- AI 建议记录写入审计日志（可追溯）
- **AI 不允许直接执行封号、删帖、驳回认证等高风险操作**
- 验收标准：admin 面板展示 AI 建议；AI 不直接执行任何裁决；建议可追溯

#### P0-12：部署 / 环境 / 监控 / 日志 / 回滚最低可生产要求

- PM2 ecosystem 覆盖 Next.js + WebSocket (`ws-server.ts`) + MCP (`mcp-server.ts`) 三个进程
- nginx WebSocket 代理配置
- Docker Compose 含 PostgreSQL + Redis（Redis 用于限流和 WS 多实例）
- 启动时环境变量自检
- 基础健康检查端点（`/api/v1/health` 已有，增加 DB/Redis 连通性检查）
- 日志输出 JSON 格式，便于采集
- 文档：部署手册、回滚流程、故障应急预案
- 验收标准：可通过 Docker Compose 一键部署；健康检查通过；日志可采集

**P0 不做项：**

- 不做推荐算法（P1）
- 不做支付宝/微信支付正式接入（P1）
- 不做 agent 绑定（P1）
- 不做 Light/Dark 主题切换
- 不做挑战赛
- 不做企业工作台重建

**P0 风险：**

1. 邮箱登录涉及发送验证邮件，需要 SMTP 服务或第三方邮件服务
2. 中国合规要求可能在实施中发现需要法律咨询的灰色地带
3. Mock 退出可能暴露此前隐藏的数据层 bug
4. 后台 AI 助手需要接入 LLM API（成本和延迟需评估）

**P0 依赖关系：**

- P0-4（合规）依赖法律咨询
- P0-5（安全基线）无外部依赖，可立即开始

- P0-11（AI 助手）依赖 LLM API 选型
- P0-12（部署）依赖 P0-3（mock 退出）和 P0-5（安全基线）

### P1：核心产品增强

> 目标：在生产落地基础上强化核心体验，形成差异化竞争力。

#### P1-1：项目画廊曝光机制

- 综合权重算法：点击量、收藏数（`ProjectBookmark`）、协作意向数、近期更新活跃度、创建者信誉分（`ContributionCredit`）
- 编辑精选（admin 手动置顶，已有 `featuredRank` 字段）
- 项目发现页增加"编辑精选"、"本周热门"、"最新发布"三个 tab
- 项目卡片增加活跃度指标展示

#### P1-2：广场推荐流 / 热度流 / 关注流 / 时间流

- 时间流：按创建时间倒序（已有）
- 热度流：按互动量（评论数 + 点赞数 `PostLike`）加权排序
- 关注流：仅展示当前用户关注的人的帖子（`UserFollow` 模型已有）
- 推荐流：基于用户兴趣标签和历史互动的简单推荐（初版用基于标签的协同过滤，不做深度学习）
- Feed tab 切换 UI

#### P1-3：Team 页协作增强

- 团队讨论区（区别于聊天，用于结构化讨论）
- 任务评论 / 任务日志
- 团队成员角色扩展：`owner | admin | member | reviewer`（当前 `TeamRole` 仅 `owner | member`）
- 团队活动时间线

#### P1-4：GitHub 仓库连接（辅助层）

- 用户可在项目设置中关联 GitHub 仓库（项目已有 `repoUrl` 字段）
- 授权后自动同步 README、Stars、Forks
- 仓库代码读取仅限用户自己授权的 agent（P2 实现 agent 侧）
- **明确边界：VibeHub 不做代码托管，不做 PR/Issue 管理，不做 CI/CD**

#### P1-5：支付宝 / 微信支付正式接入

- 实现 `AlipayProvider` 和 `WechatPayProvider`（基于 P0-9 的抽象层）
- 前提：商户号已就绪、备案已完成
- 如果商户号未就绪，先完成 sandbox/staging 联调
- 订阅付费流程：选择套餐 → 选择支付方式 → 跳转支付 → 回调确认 → 订阅生效
- 账单记录、续费提醒、到期处理、退款流程

#### P1-6：开发者 API / MCP 接入体验优化

- API 文档站点（基于已有 OpenAPI spec 生成交互式文档）
- MCP v2 manifest 自动发现
- API Key 管理界面增强（scope 可视化、用量统计）
- Quick Start 指南

#### P1-7：用户绑定多个 agent 的基础模型

- 新增 `AgentBinding` 模型：一个用户可绑定多个 agent，每个有独立 label、类型（Cursor/OpenClaw/Codex/自定义）、绑定时间
- Agent 通过独立 API Key 认证（复用 `ApiKey` 模型，增加 `agentBindingId` 字段）
- Agent 权限继承用户权限，不允许超越用户权限范围
- Agent 操作写入审计日志（`AuditLog` 增加 `agentId` 字段）

#### P1-8：管理员后台 AI 增强

- AI 预审：新帖子/新项目提交后，AI 自动进行内容安全预检，标记风险等级
- AI 摘要：对长篇举报工单自动生成摘要，提炼核心事实
- AI 打标：对企业认证资料自动比对公开信息，生成核验建议
- AI 分流：根据工单类型和紧急程度自动分配优先级
- 所有 AI 结果均为建议，不自动执行

**P1 不做项：** agent 角色体系（P2）；agent 代执行任务（P2）；挑战赛；企业工作台

**P1 风险：**

1. 推荐算法冷启动问题（初期内容少，推荐质量差）
2. 支付宝/微信支付商户号审批周期不可控
3. Agent 绑定模型设计不当可能在 P2 需要大幅重构

**P1 依赖关系：**

- P1-2（Feed 流）依赖 P0-7（广场主链路）
- P1-5（中国支付）依赖 P0-9（支付抽象层）+ 商户号就绪
- P1-7（agent 绑定）依赖 P0-5（API 安全基线）

**P1 验收标准：**

- 项目画廊有至少 3 种排序/发现机制
- 广场讨论有 4 种 feed 流可切换
- 团队页可进行结构化讨论和角色分工
- 至少一种中国支付渠道可用（或 staging 环境验证通过）
- 用户可绑定至少 1 个 agent 并通过其执行 API 调用

### P2：差异化能力建设

> 目标：构建 agent 协作闭环的核心能力，形成平台差异化壁垒。

#### P2-1：Agent 角色体系

- Agent 继承其绑定用户在团队中的角色标签
- 队长 agent、队员 agent、审查 agent 的身份映射
- Agent 可读取团队任务列表、里程碑状态

#### P2-2：Agent 协作流

- 队长 agent 可通过 API 创建任务并分配给队员
- 队员 agent 可标记任务为完成
- 完成任务后可自动通知审查角色
- 审查 agent 可提交审查意见（但最终通过/驳回需人工确认）

#### P2-3：Agent 代执行任务与审计流

- 所有 agent 操作写入独立审计表
- 高风险操作（删除、权限变更）强制人工确认
- Agent 操作限速独立于用户限速

#### P2-4：多 Agent 权限隔离

- 同一用户的不同 agent 可有不同 scope
- Agent A（Cursor）可读代码，Agent B（ChatBot）仅可读任务
- Scope 通过 API Key scopes 实现

#### P2-5：Agent 任务通知与验收闭环

- 平台内通知通道（`InAppNotification` 扩展）

## E. Agent 身份与授权模型专项设计

### 1. 用户、团队成员、agent 三者关系建模

```
User (1) ──→ (N) AgentBinding
User (1) ──→ (N) TeamMembership
AgentBinding (1) ──→ (1) ApiKey （agent 通过独立 API Key 认证）
TeamMembership 持有 teamRole（owner/admin/member/reviewer）
AgentBinding 继承其 User 的所有 TeamMembership 角色
```

- **User** 是平台的一等公民，拥有完整身份。
- **AgentBinding** 是 User 的授权代理绑定记录，每个绑定代表一个 agent 实例。
- **TeamMembership** 表达 User 在团队中的角色。Agent 不直接拥有团队角色，而是通过其绑定用户间接获得。

### 2. 一个用户绑定多个 agent 的推荐方案

新建 `AgentBinding` 模型：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String @id | |
| userId | String | 绑定用户 |
| label | String | "My Cursor Agent" |
| agentType | String | cursor / openclaw / codex / custom |
| description | String? | |
| active | Boolean @default(true) | |
| createdAt | DateTime | |
| updatedAt | DateTime | |

- 每个 AgentBinding 关联一个独立 ApiKey
- 用户可在 `/settings/agents` 管理绑定
- 每个 agent 可有不同的 API Key scope（实现能力隔离）

### 3. Agent 是"用户授权代理"，不是独立主体

理由：
- Agent 不能独立注册、不能独立持有资产、不能独立承担责任
- Agent 的所有操作法律上归属于其绑定用户
- Agent 不应有独立于用户的权限
- 中国合规基础要求：任何自动化代操作必须可追溯到具体自然人

### 4. Agent 权限继承与限制

- **继承规则：** Agent 的最大权限 ≤ 其绑定用户的当前权限
- **限制手段：** API Key scopes 进一步收窄
- **动态校验：** 每次 API 调用先验证 API Key scope，再验证绑定用户的当前权限（用户被封禁则 agent 立即失效）
- **团队权限继承：** Agent 在团队内的操作权限等于其绑定用户在该团队的角色权限

### 5. Agent 操作必须写审计日志

所有 agent 发起的 API 调用必须写入 `AuditLog`，包含：`actorId`（绑定用户 ID）、`agentBindingId`（哪个 agent）、`action`、`entityType` + `entityId`、`clientIp`、`userAgent`。通过 `agentBindingId` 是否为空区分 agent 操作与用户直接操作。

### 6. 队长 agent 派任务的合法性校验链路

1. 验证请求 API Key 对应的 AgentBinding 有效且 active
2. 验证 AgentBinding 的绑定用户在目标团队中角色为 owner 或 admin
3. 验证被分配的队员确实是该团队的成员
4. 验证 API Key scope 包含 `write:team-tasks`
5. 写入任务记录 + 审计日志
6. 通知被分配队员

### 7. 必须人工确认的 agent 行为

| 行为类别 | 是否需人工确认 | 理由 |
|---------|-------------|------|
| 读取项目/任务/帖子 | ❌ 不需要 | 纯读操作 |
| 创建任务 | ❌ 不需要 | 可撤销 |
| 标记任务完成 | ❌ 不需要 | 审查者会复核 |
| 提交审查意见 | ❌ 不需要 | 最终裁决需人工 |
| **审查通过/驳回** | ✅ 必须 | 影响工作流状态 |
| **删除任务/项目/帖子** | ✅ 必须 | 不可逆操作 |
| **团队成员移除** | ✅ 必须 | 涉及人事决策 |
| **权限变更** | ✅ 必须 | 安全敏感 |
| **发布帖子/项目** | 视配置可选 | 初期建议需确认 |

### 8. 为什么这一整套机制不能直接上 P0

1. **身份模型未验证：** AgentBinding 是全新模型，与 User、ApiKey、TeamMembership 的交互关系复杂，设计失误成本极高。
2. **团队角色体系不足：** 当前 `TeamRole` 仅有 owner/member，不支持 reviewer 等角色。Agent 协作流依赖角色体系扩展。
3. **审计基建不完善：** 当前结构化日志尚未全面铺开，在审计能力不完整的情况下上线 agent 代操作不可接受。
4. **中国合规灰色地带：** Agent 代操作的法律责任归属、授权证明要求等需专项核验。
5. **生产基线未达标：** P0 的核心任务是让平台从"可演示"到"可放量"。在 mock 未退出、安全基线未建立、登录体系未重构的情况下加入 agent 协作是"功能跑在基建前面"。

**结论：P1 做绑定模型验证，P2 做完整协作流。**

## F. 支付与订阅专项设计

### 1. 为什么中国路径下必须支持支付宝 / 微信支付

- Stripe 在中国大陆需要国际信用卡，绝大多数个人开发者没有。
- 支付宝和微信支付覆盖中国 95% 以上的在线支付场景。
- 不支持支付宝/微信支付等于没有收款能力。

### 2. 为什么不能把个人收款码视为正式方案

- 无法获取支付回调（无法自动确认订单）
- 无法实现自动续费
- 无法开具发票
- 存在合规风险（经营性收入通过个人账户收取）
- 微信/支付宝可能冻结高频收款的个人账户

### 3. 推荐的支付抽象层设计

```typescript
interface PaymentProvider {
  createCheckoutSession(params: CheckoutParams): Promise<CheckoutSession>;
  handleWebhook(payload: Buffer, signature: string): Promise<WebhookEvent>;
  createPortalSession(customerId: string): Promise<PortalSession>;
  cancelSubscription(subscriptionId: string): Promise<void>;
  getSubscription(subscriptionId: string): Promise<SubscriptionInfo>;
}
```

- `UserSubscription` 增加 `paymentProvider` 字段（`stripe` / `alipay` / `wechatpay`）
- 路由层根据用户选择的支付方式分派到对应 provider
- Webhook 路由按 provider 分别处理签名验证

### 4. Free / Pro 与中国支付链路对接

- Free 用户无需支付
- Pro 订阅：选择套餐 → 选择支付方式 → 对应 provider 创建会话 → 支付 → 回调确认 → 订阅生效
- 中国渠道的自动续费开通率较低，建议同时支持手动续费提醒

### 5. 支付生命周期设计

| 场景 | 处理逻辑 |
|------|---------|
| 支付成功 | Webhook → `UserSubscription.status = active` → 设置 `currentPeriodEnd` → 通知用户 |
| 自动续费成功 | 同上，更新 `currentPeriodEnd` |
| 续费失败 | `status = past_due` → 通知 → 宽限期 7 天 → 降为 free |
| 退款 | 管理员发起 → provider.refund() → `status = canceled` → 降为 free |
| 取消订阅 | `cancelAtPeriodEnd = true` → 到期后降为 free |
| 账单记录 | 新建 `BillingRecord` 模型记录每笔交易 |

### 6. Staging / Sandbox 占位方案

- P0：支付抽象层代码就位；Stripe sandbox 可用；支付宝/微信 provider 类骨架代码就位；UI 上中国渠道标注"即将上线"
- P1：商户号就绪后正式切换

### 7. 支付模块优先级判定

- **P0**：支付抽象层架构 + `UserSubscription` 改造 + Stripe 迁移到抽象层。理由：不做就无法在 P1 接入中国渠道。
- **P1**：支付宝/微信支付正式接入 + 订阅闭环细化 + 账单记录。理由：依赖商户号就绪。

## G. 管理员后台与 AI 治理助手专项设计

### 1. 管理员后台核心职责

1. 用户管理：查看列表、封禁/解封、角色变更
2. 内容审核：帖子审核（pending → approved/rejected）、评论管理
3. 举报处理：工单查看、分类、处理、关闭
4. 企业认证审核：申请查看、资料核验、审批/驳回
5. 协作审核：协作意向审核
6. 审计日志：操作日志查看、筛选、导出
7. 系统监控：健康状态、关键指标、队列状态

### 2. 最适合 AI 辅助的任务

| 任务 | AI 适合度 | 理由 |
|------|----------|------|
| 举报工单摘要 | ⭐⭐⭐⭐⭐ | 大量文本快速理解 |
| 内容安全预检 | ⭐⭐⭐⭐⭐ | 敏感词/违规模式检测 |
| 企业认证资料比对 | ⭐⭐⭐⭐ | 信息一致性检查 |
| 用户行为模式识别 | ⭐⭐⭐⭐ | 批量数据异常模式 |
| 工单优先级分流 | ⭐⭐⭐ | 规则 + AI 混合 |
| 内容质量评估 | ⭐⭐⭐ | 主观性较强 |
| 封号/删帖决策 | ⭐ | 高风险，必须人工 |

### 3. AI 后台可做事项

- **摘要**：举报工单、长帖子、企业认证资料的结构化摘要
- **打标**：内容风险等级（低/中/高）、类别（广告/色情/政治/版权/其他）
- **风险分级**：综合用户历史行为、内容特征、举报频率的风险评分
- **预审建议**：帖子审核、企业认证的"建议通过"/"建议驳回"/"建议人工详审"
- **工单分流**：根据类型和紧急程度自动分配优先级
- **资料比对**：企业名称/网站/行业信息交叉核验
- **内容巡检建议**：定期扫描，生成违规风险报告

### 4. 必须人工最终确认的高风险动作

| 动作 | 风险等级 | 必须人工确认 |
|------|---------|------------|
| 封禁用户 | 🔴 极高 | ✅ |
| 删除帖子/评论 | 🔴 高 | ✅ |
| 驳回企业认证 | 🔴 高 | ✅ |
| 关闭举报工单 | 🟠 中 | ✅ |
| 变更用户角色 | 🔴 极高 | ✅ |
| 取消用户订阅 | 🔴 高 | ✅ |
| 批量操作 | 🔴 极高 | ✅ + 二次确认 |

### 5. AI 后台避免越权

- **架构隔离**：AI 服务仅有读权限 + 建议写入权限，不持有 mutating API 凭证
- **API 层面**：高风险操作端点强制要求 `adminUserId` 参数（自然人身份）
- **UI 层面**：AI 建议以独立卡片展示，与"确认执行"按钮物理分离
- **审计层面**：AI 建议和管理员最终决策分别记录

### 6. AI 建议审计、追溯、复核

新建 `AdminAiSuggestion` 模型：

| 字段 | 说明 |
|------|------|
| id | 唯一 ID |
| targetType | report_ticket / enterprise_verification / post_review |
| targetId | 目标实体 ID |
| suggestion | AI 建议内容 |
| riskLevel | low / medium / high |
| confidence | AI 自评置信度 |
| adminDecision | accepted / rejected / modified |
| adminUserId | 处理的管理员 |
| decidedAt | 决策时间 |
| createdAt | 建议生成时间 |

### 7. 三类场景设计

**企业认证审核：** AI 比对企业名称公开记录 → 检查网站域名一致性 → 生成"资料完整度评分"和"建议操作" → 管理员查看 AI 摘要 + 原始资料 → 人工审批/驳回

**举报工单：** AI 摘要举报内容和被举报内容 → 评估违规概率和严重程度 → 建议处置方式 → 管理员查看 → 人工处置

**内容审核：** AI 对新帖子预检风险等级 → 低风险进入"建议通过"队列 → 高风险进入"需人工审核"队列 → 管理员处理

### 8. 为什么 AI 必须是"治理助手"而非"自动执法者"

1. **法律责任**：平台对内容审核的法律责任不能委托给 AI
2. **误判成本**：封禁无辜用户、删除合规帖子的伤害不可逆
3. **上下文局限**：AI 无法理解所有社区语境
4. **合规要求**：中国相关法规明确要求平台承担主体责任（需专项合规核验）
5. **用户信任**：治理公信力依赖人工兜底

| 数据跨境 | GitHub API 数据是否构成出境？Stripe 跨境？ | P0 核验 |
| 日志留存期限 | 审计日志留存多久？不同类型差异？ | P0 核验 |
| AIGC 服务备案 | AI 辅助功能是否需备案？ | P1 核验 |
| 算法推荐备案 | 推荐流/热度流是否需备案？ | P1 核验 |
| ICP 备案 | 域名和服务器 ICP 备案 | P0（阻塞上线） |
| 支付商户资质 | 支付宝/微信商户号审批要求 | P1 核验 |
| Agent 代操作授权证明 | 授权记录的法律效力和留存要求 | P2 核验 |
| 未成年人保护 | 是否需年龄验证/使用限制？ | P0 核验 |

### 上线前必须确认项

| 项目 | 确认内容 |
|------|---------|
| ICP 备案完成 | 域名已备案，服务器在中国大陆 |
| 隐私政策经法律审核 | 内容符合《个人信息保护法》要求 |
| 用户协议经法律审核 | 包含必要免责条款和用户权利说明 |
| 内容审核机制就位 | 人工审核流程 + AI 辅助预检可用 |
| 举报处理时效承诺 | 明确举报处理时限 |
| 数据存储方案确认 | 服务器在中国大陆或确认跨境合规 |
| 支付方案合规 | 收款通过正规商户通道 |
| AI 后台责任边界 | AI 仅辅助，人工兜底到位 |

## I. 最终执行顺序

**Step 1：安全基线与 Mock 退出**（P0-3, P0-5）
- 生产环境 mock 退出
- Session 吊销、CSRF、Admin 边缘验证
- 结构化日志替换 `console.*`
- 启动时环境变量自检

**Step 2：登录体系重构**（P0-1）
- 邮箱注册/登录
- GitHub 降为关联
- 密码哈希存储、邮箱验证流

**Step 3：合规框架与法律文件**（P0-4）
- 隐私政策、用户协议内容填充
- 注册勾选同意、账号注销功能
- 并行启动：ICP 备案、实名制核验、数据跨境咨询

**Step 4：企业能力降级**（P0-2）
- EnterpriseProfile 降级为认证标识
- 移除工作台入口
- 后台保留审核

**Step 5：核心三大模块主链路收口**（P0-6, P0-7, P0-8）
- 项目画廊编辑页补全
- 广场讨论全链路确认
- 团队设置页补全
- 可并行执行

**Step 6：支付抽象层 + 订阅改造**（P0-9）
- PaymentProvider 接口设计与实现
- Stripe 迁移到抽象层
- UserSubscription 增加 paymentProvider 字段

**Step 7：管理员后台重构 + AI 审核助手 MVP**（P0-10, P0-11）
- 后台面板补全
- AI 审核助手 MVP（摘要 + 风险标记 + 预审建议）
- LLM API 选型与接入

**Step 8：部署与运维生产化**（P0-12）
- PM2 全进程管理
- Docker Compose 含 Redis
- nginx WebSocket 代理
- 健康检查完善
- 部署文档

**— P0 完成线 —**

**Step 9：Feed 流与曝光机制**（P1-1, P1-2）
- 项目画廊曝光权重算法
- 广场四种 Feed 流
- 推荐算法初版

**Step 10：团队协作增强 + GitHub 辅助连接**（P1-3, P1-4）
- 团队角色扩展
- 任务评论/日志
- GitHub 仓库连接与 README 同步

**Step 11：支付宝/微信支付正式接入**（P1-5）
- 前提：商户号就绪
- 如未就绪先完成 sandbox 联调

**Step 12：Agent 基础绑定模型 + API 体验优化**（P1-6, P1-7）
- AgentBinding 模型设计与实现
- API 文档站
- 用户可绑定 agent

**Step 13：后台 AI 增强**（P1-8）
- 预审、摘要、打标、分流能力增强

**— P1 完成线 —**

**Step 14：Agent 角色体系与协作流**（P2-1 ~ P2-5）
- Agent 角色映射
- 任务分配与通知
- 审查流
- 审计与权限隔离

**Step 15：AI 后台复杂工单升级**（P2-6）
- 批量处理、行为模式分析、内容巡检报告

**— P2 完成线 —**

**Step 16+：生态扩展**（P3）
- 根据数据指标决定优先级
- 挑战赛、多客户端、agent 编排、信誉体系等按需启动
