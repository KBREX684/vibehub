# VibeHub 正式上线标准（冻结版）

更新日期：2026-04-16
状态：Frozen Standard v1
用途：作为 VibeHub 是否允许进入正式上线（Go-Live）的唯一判定基线；后续任何新增需求、规划、返工、验收，均不得绕开本标准。

---

## 0. 目的

这份标准用于解决两个问题：

1. 防止项目范围无限膨胀，导致“永远在做功能、始终不上线”。
2. 防止在缺少生产条件的情况下，为了进度或情绪而强行上线。

判定原则：

- **只有满足 P0 上线硬门槛，项目才允许进入正式上线准备。**
- **只有满足“上线判定清单”的全部必过项，项目才允许被判定为“可正式上线”。**
- **任何新需求不得阻断上线，除非它命中本标准中的“硬阻塞项”。**

---

## 1. 产品边界冻结

正式上线版本的 VibeHub，只允许围绕以下主飞轮：

**社区讨论 -> 项目曝光 -> 协作意向 / 团队组建 -> 团队任务推进 -> 开发者 API / MCP 接入**

上线前产品边界固定如下：

### 1.1 必须存在的核心能力

1. 登录与账号体系
   - 邮箱登录
   - GitHub 登录 / GitHub 账号关联
   - 基础账号安全能力

2. 社区讨论
   - 发帖、列表、详情、评论、回复
   - 基础推荐 / 热度 / 时间 / 关注流中的最小闭环
   - 举报与审核

3. 项目画廊
   - 项目创建、编辑、详情、列表、搜索 / 筛选
   - 收藏 / 点击 / 互动数据可形成基础曝光排序

4. 团队协作
   - team 创建
   - 入队申请 / 审核
   - 团队任务 / 里程碑最小闭环
   - 团队页内讨论 / 聊天最小闭环

5. 开发者接入
   - API Key
   - OpenAPI
   - MCP 基础接入
   - 至少支持用户与 agent 的基础绑定模型

6. 商业化最小闭环
   - Free / Pro
   - 支付抽象层具备可落地能力
   - 中国支付路径至少完成正式接入方案与上线所需配置闭环

7. 管理员后台
   - 用户治理
   - 内容治理
   - 举报工单
   - 企业认证审核
   - 后台 AI 助手仅做辅助，不做最终高风险裁决

### 1.2 明确不上线版本的能力

以下能力**不得作为当前正式上线阻塞项**，也不得抢占当前主线资源：

- 挑战赛 / 活动系统
- 复杂企业工作台
- 多层企业协同产品
- 高复杂度 agent 自动自治网络
- 无人类确认的高权限 agent 自动执行
- 大型增长玩法系统
- 非核心视觉炫技改版

## 2. 正式上线硬门槛（P0 Gate）

以下任意一项不满足，项目即 **不得正式上线**。

### 2.1 合规与账号

必须满足：

- 邮箱登录可用，登录、登出、找回、基础验证链路完整。
- GitHub 登录不是唯一入口，只能作为辅助入口。
- 用户协议、隐私政策、基础平台规则可访问。
- 举报、审核、封禁 / 限制的最小治理闭环存在。
- 企业认证若上线，只能作为“认证信息 / 身份标识”，不能重新成为产品中心。
- 所有涉及中国合规边界的不确定项已被单独列出并完成专项核验或标记为上线前必确认项。

### 2.2 生产环境隔离

必须满足：

- 生产环境禁止隐式 mock fallback。
- staging / production 环境变量分离清晰。
- 必要 secrets 已有独立配置，不依赖默认开发值。
- 生产环境配置缺失时，系统必须 fail fast，而不是带病运行。

### 2.3 安全与权限

必须满足：

- 登录态、CSRF、会话签名、权限判断闭环有效。
- 管理员后台具备独立权限边界。
- API Key 支持 scopes、撤销、最小权限控制。
- 用户与 agent 的绑定关系有明确权限边界。
- 高风险 agent 行为必须有人类确认。
- 核心高风险操作有审计日志。

### 2.4 核心业务闭环

必须满足：

- 讨论区：列表 -> 发帖 -> 评论 -> 举报 / 审核闭环可用。
- 项目画廊：创建 -> 展示 -> 互动 -> 曝光排序最小闭环可用。
- 团队：创建 -> 入队 -> 任务推进 -> 协作沟通闭环可用。
- API / MCP：至少一个真实接入主链路通畅。
- Free / Pro：套餐状态能真实生效。

### 2.5 支付与订阅

必须满足：

- 支付体系不能停留在个人收款码层级。
- 支付抽象层已设计并落地到可扩展结构。
- 至少明确并完成中国主流支付路径的生产方案：支付宝 / 微信支付。
- 订阅状态、开通、取消、失败、账单记录有明确状态机。
- 在商户、备案、正式参数未齐备前，不得将“伪支付”冒充正式支付上线。

### 2.6 运维与可观测性

必须满足：

- 健康检查可用。
- 关键错误日志可追踪。
- 关键操作具备 request id / trace 基础能力。
- 有最小告警机制或可替代人工巡检流程。
- 有明确回滚路径。
- 数据库迁移可重复执行且 CI 可验证。

### 2.7 质量基线

必须满足：

- CI 通过。
- 核心主链路 smoke 通过。
- 核心 E2E 通过。
- OpenAPI 与生成类型一致。
- 无已知 P0 级数据破坏风险。
- 无已知 P0 级鉴权绕过风险。

## 3. 上线判定清单（全部必过）

只有当以下问题的答案全部为“是”，才允许判定为“可正式上线”：

### 3.1 产品问题

- 是否已经有唯一主飞轮，而非多中心产品？
- 是否已经冻结非主线需求，不再无边界扩张？
- 是否已经明确挑战赛、复杂企业能力、多 agent 自治网络不属于当前上线阻塞项？
- 是否已经能让一个真实用户完成：注册 / 登录 -> 浏览讨论 -> 浏览项目 -> 加入团队 / 参与协作 -> 使用开发者接入能力？

### 3.2 用户与社区问题

- 是否已有最小内容治理闭环？
- 是否已有举报入口与后台处理能力？
- 是否已有关注 / 时间流 / 热度流中的最小可用方案？
- 是否已有项目曝光排序的最小可信机制？

### 3.3 团队协作问题

- 是否可以真实创建 team？
- 是否可以申请加入并审核？
- 是否可以分配任务并推进？
- 是否可以进行团队内沟通？
- 聊天 / 讨论能力是否至少在单实例生产部署下稳定？

### 3.4 接入生态问题

- API Key 是否可以生成、撤销、限制权限？
- MCP 是否可以被真实客户端接入？
- 一个用户是否已具备绑定一个或多个 agent 的基础结构？
- agent 是否仍然处于“用户授权代理”而非“独立高权限主体”？

### 3.5 支付问题

- Free / Pro 是否已真实区分？
- 支付状态是否能驱动订阅状态变化？
- 是否已有中国支付路径可落地方案？
- 是否避免了“个人收款码假装 SaaS 支付系统”的错误路线？

### 3.6 后台治理问题

- 管理员后台是否能处理用户、内容、举报、认证等核心事务？
- 后台 AI 是否仅提供建议、摘要、打标、预审，而非越权裁决？
- 所有 AI 辅助结论是否可追溯、可复核？

### 3.7 技术问题

- 生产是否已完全脱离隐式 mock？
- 是否具备最小分布式或可扩展设计意识（限流 / 日志 / 配置）？
- 是否具备回滚能力？
- 是否具备环境隔离能力？
- 是否具备最小可观测性？

如果以上任一项答案为“否”，则结论必须是：

**暂不可正式上线。**

## 4. 范围控制规则（防止后续任务无限膨胀）

从本标准生效起，后续所有新增需求，必须先经过以下四问：

1. 这个需求是否直接增强主飞轮？
2. 这个需求是否属于正式上线硬门槛？
3. 这个需求是否会阻塞当前上线？
4. 这个需求是否可以放入 P2 / P3？

判定规则：

- 若答案为“不是主飞轮增强、不是硬门槛、但实现复杂度高”，则必须后移。
- 若答案为“吸引人但不影响上线”，则必须后移。
- 若答案为“会显著增加合规 / 支付 / 权限 / 运维复杂度”，则必须重新评估，不得直接进入当前冲刺。

### 4.1 后续任务只允许进入四类之一

- **P0 硬阻塞修复**：不上线就不行
- **P1 上线增强项**：提升首发质量
- **P2 差异化能力**：有价值，但不阻塞首发
- **P3 中长期扩展**：明确后移

### 4.2 明确后移列表（默认不进入当前上线范围）

- 挑战赛 / 活动页
- 重企业化工作台
- 多层级复杂组织系统
- 全自动 agent 审批网络
- 无人工确认的 agent 执行链
- 非必要 UI 大重构
- 与主飞轮无关的增长玩法

---

## 5. “可正式上线” 的唯一结论条件

当且仅当同时满足以下条件，VibeHub 才能被判定为：

**可正式上线（Go-Live Ready）**

### 5.1 必须同时满足

1. P0 Gate 全部通过
2. 上线判定清单全部为“是”
3. 当前无已知 P0 级安全风险
4. 当前无已知 P0 级数据一致性风险
5. 当前无已知 P0 级支付 / 权限 / 合规硬阻塞
6. 当前产品范围已被冻结，不再继续扩张当前首发范围
7. 后台治理与人工兜底机制已就位

### 5.2 一票否决项

以下任一项存在，则一票否决：

- 生产仍可能隐式走 mock
- 支付方案仍是假支付 / 不可追踪
- 后台无治理闭环
- 高风险 agent 行为无人工确认
- 合规必需项未确认
- 核心权限存在已知绕过风险
- 上线后无回滚能力

## 6. 后续执行要求

从现在开始，后续任何规划、验收、需求拆解，都必须遵守以下输出规则：

1. 必须先判断该事项属于 P0 / P1 / P2 / P3 哪一级。
2. 必须先判断它是否影响正式上线标准。
3. 若不影响正式上线，不得冒充“当前必须做”。
4. 若达到本标准，必须明确输出：
   - **“VibeHub 已达到正式上线标准。”**
5. 若未达到，必须明确输出：
   - **“VibeHub 当前尚未达到正式上线标准。”**
   - 并列出缺失项。

---

## 7. 推荐使用方式

以后每次做以下事情时，都必须引用本标准进行判定：

- 做新需求规划
- 做返工单
- 做阶段验收
- 做上线前检查
- 做 Claude / Cursor / GPT 项目经理提示词

标准用法：

### 7.1 规划时
先问：
- 这个需求是否属于正式上线标准中的硬门槛？

### 7.2 验收时
先问：
- 这个结果是否让我们更接近“Go-Live Ready”？

### 7.3 决策时
先问：
- 这个需求是否会导致上线范围膨胀？

每次阶段性复盘，都必须在结尾给出以下二选一结论：

**VibeHub 当前尚未达到正式上线标准。**
缺失项：
- xxx
- xxx
- xxx

### 结论 B
**VibeHub 已达到正式上线标准。**
允许进入正式上线准备与最终发布流程。

## 9. 备注

本标准优先级高于临时灵感、短期冲动需求、单次对话中的扩张性建议。
若后续规划与本标准冲突，以本标准为准，除非明确发布新版冻结标准。
